# MDSDRV / ctrmml 互換メモ

この文書は、`ctrmml` 実装を MKVDRV-Wasm に移す際に、互換性上の注意点を短く整理したものです。

## 1. 仕様書より実装を優先すべき箇所

`ctrmml/mml_ref.md` は全体像を掴むには有用ですが、最終的には `mml_input.cpp`, `track.cpp`, `test_mml_input.cpp` を優先したほうが安全です。

理由:

- 仕様書にあるが、現実装で未対応のものがある
- 逆に、実装固有の扱いがある
- テストが実際の期待挙動を最も強く示す

## 2. `%` の扱い

`mml_ref.md` には `%` を「direct length」と読む文脈がありますが、`ctrmml/src/mml_input.cpp` の現実装では本文中の `%` は `Event::PLATFORM` に数値を積む処理です。

このため、MKVDRV-Wasm 側では `%` の意味を早い段階で固定する必要があります。

推奨:

- MDSDRV-mml 互換を優先するなら `%` の実際の用法を追加調査
- `ctrmml` 互換を優先するなら、まず現実装通り `PLATFORM` 扱いで受ける

## 3. `o<num>` の内部オクターブ

`o<num>` は `set_octave(num - 1)` で処理されます。見た目と内部表現が 1 ずれるため、Rust 実装では「MML 上の octave」と「内部 note base」を分けて持ったほうが安全です。

## 4. `h` は `b` と同義

ドイツ音名系の `h` がそのまま受理されます。`b` と別扱いにしないこと。

## 5. 条件ブロックは単なる AST 分岐ではない

`{.../...}` は「複数トラックに対して同じ行を個別に再パースし、`track_offset` で分岐位置を決める」実装です。

このため、単純な 1 回 parse + 後段分配では挙動が変わる可能性があります。

## 6. 後方修正コマンドが多い

次のコマンドは、すでに積んだイベントを後から書き換えます。

- `^`
- `&`
- `R`
- `~`
- `\`

特に `R` と `~` は reverse backtracking が必要です。ストリーム出力型より、編集可能な event vector の方が実装しやすいです。

## 7. デフォルト量子化

既定では `Q8` 相当で、ノートはフル長発音です。`q` を使った場合だけ末尾休符が生じます。

つまり、Wasm 側で簡易シーケンスを実装するときの「gate ratio デフォルト」は 100% に近い挙動を基準にした方が、ctrmml 互換に寄せやすいです。

## 8. echo は音量イベントも含めて再現する必要がある

echo は単なる過去ノート再利用ではありません。

- delay 参照
- buffer clear / retain
- volume 差分の前後挿入
- 休符の再利用

まで含めて仕様です。

## 9. MML 仕様書の配置方針

今後 `_reference/mml_spec` には最低限、以下を維持するのが望ましいです。

- `commands.md`
- `state_rules.md`
- `examples.md`
- `compatibility_notes.md`

この 4 つを更新しながら Rust テストを追加していくと、AI 実装でも人手実装でも挙動ぶれを抑えやすいです。
